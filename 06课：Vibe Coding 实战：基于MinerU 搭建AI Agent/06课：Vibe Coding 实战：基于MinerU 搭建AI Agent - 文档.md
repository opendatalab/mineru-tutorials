**一、教学说明**

这节课最重要的目标，不是让大家记住多少代码细节，而是让大家看到一件更有价值的事情：

**即使你不熟悉代码，也可以借助 AI，从一个想法出发，快速推进一个真实项目。**

这门课想解决的问题，不只是"怎么做一个功能"，而是：

如何把一个模糊想法说清楚

如何让 AI 帮你整理需求和计划

如何让 AI 帮你推进开发和调试

如何让 AI 帮你完成测试、本地启动演示和项目整理

也就是说，本节课的主线不是"讲架构"，而是"讲过程"：

**如何用 AI，把一个项目从想法推到可运行、可展示、可分享。**

**二、学习目标**

学完这节课后，同学们应当能够：

理解什么是 Vibe Coding

知道如何向 AI 清楚描述一个项目目标

学会让 AI 帮你拆需求、做计划、改代码、修问题

学会在不懂全部代码的情况下推进项目

了解如何把 AI 产出的结果整理成可演示、可交付的项目成果

**三、什么是 Vibe Coding**

这里的 Vibe Coding，不是随便"让 AI 写代码"。

它更像是一种项目推进方式：

你负责提出目标

你负责判断结果对不对

AI 负责帮你整理、生成、修改、补全

在这个过程中，人并没有消失，而是从"逐行写代码的人"变成了：

任务定义者

结果判断者

迭代推进者

所以，对初学者来说，Vibe Coding 最重要的能力不是编程语法，而是：

会描述问题

会拆分任务

会给 AI 反馈

会判断结果

**四、适合哪些同学学习**

本教材适合以下同学：

想用 AI 快速上手项目的同学

代码基础还不够强，但希望参与真实项目的同学

想学习如何把"需求 -\> AI -\> 成果"串起来的同学

需要准备课程展示、比赛展示或结课项目的同学

**六、导入：这节课到底要解决什么问题**

在原有 NanaDraw 中，用户主要通过文字描述来生成图示。

但在真实使用中，很多时候用户手里已经有论文或技术文档。这时用户真正想做的事情并不是重新写一遍说明，而是：

直接上传 PDF

让系统先解析文档

从文档里选出关键片段

再根据这些内容生成图示

所以，这节课的核心不是"做一个上传按钮"，而是：

**让文档内容真正进入 AI 绘图流程。**

同时，这节课还有第二条更重要的暗线：

**这个功能不是靠一个人纯手写完成，而是通过 AI 协助，一步步推进出来的。**

**第 1 部分 需求 Spec**

*这一部分可以基本保持不变，因为无论是不是 Vibe Coding，第一步都必须把需求说清楚。*

**1.1 本部分学习目标**

在这一部分，同学们需要明白：

为什么不能只说一句"支持上传 PDF"

为什么需求越清楚，AI 越能帮得上忙

**1.2 一个模糊需求的问题**

比如一句话：

"希望 NanaDraw 支持上传 PDF，并根据论文内容来生成图。"

这句话方向是对的，但还不够落地。因为它至少没有回答下面几个问题：

从哪里上传

上传什么格式

解析结果怎么展示

用户怎么选内容

选中的内容怎么接回绘图流程

如果这些问题不先说清楚，后面无论是人写代码，还是 AI 帮忙写代码，都会越做越乱。

**1.3 在本项目中，需求是怎样拆开的**

结合 NanaDraw 和 MinerU 的能力，我们把需求拆成：

在 AI 助手区域提供 PDF 上传入口

在设置中配置文档解析 Token

后端调用 MinerU 完成 PDF 解析

解析后展示完整文本

用户可以选择文本片段作为绘图参考

选中内容后，继续驱动 NanaDraw 原有生成流程

**1.4 为什么要先做 Spec，再找 AI**

这是同学们最需要记住的一点：

AI 不是魔法。如果你自己都说不清要做什么，AI 也很难给出稳定结果。

所以，Spec 的作用就是把一个模糊想法整理成 AI 能理解、你也能检查的任务描述。

**1.5 本部分小结**

第一步不是写代码，而是把问题定义清楚。对于 Vibe Coding 来说，这一步尤其重要，因为：

**你给 AI 的描述越准确，AI 给你的结果才越可靠。**

**第 2 部分 用 AI 做 Plan**

**2.1 本部分学习目标**

在这一部分，同学们要学会的不是系统架构细节，而是：

如何让 AI 帮你把需求变成计划

如何把一个项目拆成几个可执行的小任务

如何让 AI 先给出"第一版最小可行方案"

**2.2 为什么第二步要先让 AI 做计划**

很多初学者一上来就直接问 AI："帮我把这个功能写出来。"

这样的问题通常太大，AI 给出的答案容易：

太散

太大

太理想化

不贴合当前项目

更有效的做法是先让 AI 做 Plan。

也就是先问它：

这个项目应该拆成哪几步

哪些功能是第一版必须做的

哪些可以后面再优化

哪些地方最容易出问题

**2.3 可以怎样向 AI 提问**

下面是一个适合课堂展示的提问方式：

**示例提示词**

*这是一个已有项目 NanaDraw，主要用于学术绘图场景，我希望增加 PDF 解析功能。用户上传 PDF 后，系统通过 MinerU 解析文档，再让用户从解析文本中选择片段生成图。请你不要直接写代码，先帮我输出：*

*相关素材：*

**2.4 AI 在这一阶段帮我们的是什么**

AI 在 Plan 阶段最有价值的，不是"替你做决定"，而是"帮你整理思路"。

它能帮助你：

把需求变得更具体

把大任务拆成小任务

提前暴露容易忽略的问题

给出一个可以照着推进的顺序

对于不懂代码的同学，这一步尤其重要。因为一旦计划出来了，你就不再面对一团模糊的东西，而是面对一组可以逐项推进的小目标。

**2.5 本部分小结**

Vibe Coding 的第二步不是"让 AI 开始写"，而是：

**先让 AI 帮你把路画出来。**

**第 3 部分 用 AI 推进 Build / Test / 本地演示**

**3.1 本部分学习目标**

这是整节课最核心的一部分。在这一部分，同学们要理解的是：

不懂代码的人，怎样也能借助 AI 推进开发

遇到错误时，怎样把问题交给 AI 继续解决

为什么 Build 和 Test 都可以成为 AI 协作的一部分

为什么本地启动和演示也可以成为 AI 协作的一部分

**3.2 Build 阶段，不要一次让 AI 做完整个项目**

这是初学者最容易踩的坑。

如果你一次就说：

"请把这个功能完整做出来。"

AI 很可能会给你一大段看起来很厉害、但很难真正落地的内容。

更好的做法是分轮推进，例如：

**第 1 轮**

让 AI 先读项目，告诉你：

当前项目相关文件在哪里

这个功能可能要改哪些部分

先做哪个点最稳

**第 2 轮**

让 AI 只做一个小目标，例如：

先加上传入口

先加设置项

先打通后端解析接口

**第 3 轮**

再让 AI 继续做下一个目标，例如：

展示完整文本

添加文本选择交互

接回生成流程

这样做的好处是：

结果更容易检查

出错时更容易定位

你不会被一大堆代码压住

**3.3 不懂代码时，最重要的是会反馈**

很多同学会觉得："我不会写代码，那我怎么判断 AI 写得对不对？"

答案是：

你不一定要先会写，但你一定要会反馈。

比如你可以这样告诉 AI：

"这个按钮位置不对"

"解析结果被截断了"

"我想要完整文本，不要摘要"

"这个按钮不要关闭，要改成收起"

"不要在界面里体现 MinerU 这个名字"

这些反馈都不是代码语言，但都是非常有效的产品语言。AI 最擅长的，就是把这种自然语言反馈转成实现修改。

**3.4 出现报错时，怎样继续用 AI**

在 Vibe Coding 中，报错不是结束，而是下一轮输入。

例如你可以把报错原样发给 AI，并补充一句：

"请解释这个报错是什么意思"

"请定位可能是前端还是后端问题"

"请给出最小修改方案"

"请修掉这个问题，不要影响已有功能"

这一步很关键，因为它体现了 AI 在项目中的另一个角色：

**不仅能帮你生成，还能帮你调试。**

**3.5 Test 阶段，AI 也可以继续参与**

很多同学以为测试只能由会写代码的人完成，其实不是。

AI 在测试阶段可以帮助你做三件事：

帮你列测试清单

帮你补测试代码

帮你设计手工验收步骤

例如你可以这样问：

*这个功能已经完成，请帮我列出：*

这样一来，即使你不熟悉测试框架，也能先形成清楚的检查思路。

**3.6 本地启动演示，也可以交给 AI 帮你推进**

很多同学做到这一步时，功能其实已经差不多完成了，但项目还没有真正"跑起来"。

这时最常见的问题不是代码本身，而是：

不知道前后端分别怎么启动

不知道需要准备哪些依赖

不知道先看哪里确认项目有没有跑通

不知道演示时应该按什么顺序操作

这时候，AI 很适合继续做"本地启动助手"。

例如你可以这样问 AI：

*这是一个前后端项目，请先不要讲上线流程，先告诉我：*

这样做的价值在于，即使你还不熟悉工程环境，也能在 AI 的帮助下，把项目整理到"可以现场展示"的状态。

**3.7 这一部分真正想教给同学们什么**

这一部分最重要的，不是记住某个接口名字，而是记住一种推进方式：

不要求 AI 一次做完

一次只推进一个小目标（todo_list）

出现问题就把结果继续喂给 AI

用自然语言做反馈

用测试和演示来验证结果

**3.8 本部分小结**

Build 和 Test 的核心，不是你会不会从零手写代码，而是：

**你能不能借助 AI，把一个功能一轮一轮推到可用状态。**

**第 4 部分 用 AI 整理 GitHub 交付**

**4.1 本部分学习目标**

在这一部分，同学们要看到：

项目完成后，AI 还可以继续帮助你整理成果

GitHub 不只是传代码，还包括分支、说明、总结和展示

在这节课里，最后一步是把成果整理后提交到 dev 分支，并发起 PR

**4.2 AI 在交付阶段最有价值的地方**

在这个课程案例里，第 4 步是把当前成果整理好，提交到 GitHub 的 dev 分支，并发起一个 PR。

当功能做完以后，很多同学会卡在最后一步：

不知道怎么写变更说明

不知道怎么写提交信息

不知道怎么写 README

不知道怎么给别人介绍这个项目

这时，AI 特别适合做"整理器"。

例如你可以让 AI：

根据改动生成适合提交到 dev 分支的变更说明

帮你写 PR 描述

帮你整理 README 的新增部分

帮你生成项目展示用的一段简介

**4.3 为什么这一步很重要**

因为项目真正的结束，不是"功能能跑了"，而是：

**别人能够看懂、接手、复用。**

而 AI 很擅长把零散的工程内容整理成更清楚的表达。

对于不会写技术文档的同学来说，这一步尤其有帮助。

**4.4 本部分小结**

GitHub 交付阶段，AI 的角色从"开发助手"变成了"表达助手"。

它帮助你把已经完成的工作，整理成别人能理解的成果。

**4.5 产品经理的实践心得**

当有一些 feature 需要开发的时候，我会采用前面几步的方式，快速实验出一个 demo，等调到我满意了，再提交一个 PR 到 GitHub 或 GitLab。PR 内容需要包括代码、feature 的变更说明等内容，然后再由开发统一集成。

先把需求说清楚

然后分轮推进开发

出现问题就继续反馈给 AI

所以，Vibe Coding 真正教给我们的不是"偷懒"，而是：

**用更高效的方式，把注意力放在目标、判断和迭代上。**

**附录：完整教材正文**

**用 AI 快速上手一个真实项目**

**以 NanaDraw + MinerU 为例，理解什么是 Vibe Coding**

在很多同学的印象里，做项目往往意味着两件事：第一，要先学很多代码；第二，要把前后端、接口、部署这些内容都弄明白，才能真正开始。

但今天的实际情况已经不太一样了。

随着 AI 编程工具越来越成熟，一个项目的推进方式正在发生变化。对于初学者来说，最重要的能力不再只是"会不会从零写完整代码"，而是能不能把自己的目标说清楚，能不能把任务拆开，能不能和 AI 协作，把一个模糊想法一步步推进成可以运行、可以演示、可以交付的成果。

这种工作方式，常常被大家叫作 Vibe Coding。

这篇文章想带同学们一起做的，不是去死记某个框架的语法，而是通过一个真实案例，理解怎样借助 AI 推进项目。

我们选择的案例是我们实际再做的一个真实项目：

在NanaDraw中集成MinerU

目标功能是：

用户上传 PDF 后，系统先解析文档内容，再从中选择合适的片段作为绘图参考，帮助生成图示。

这个案例很适合教学，因为它既有清楚的用户需求，又包含前端交互、后端处理、错误修正、本地启动和结果交付，基本涵盖了一个小型项目从想法到结果的完整路径。

**一、这篇文章想解决什么问题**

很多同学会问：

"我代码基础一般，甚至不太会写代码，那我还能不能做项目？"

答案是，可以，但前提是你要学会一种新的推进方式。

过去，项目开发更像是一种"手工完成"的过程。你需要自己写代码、自己排查问题、自己整理结果。现在，AI 可以承担其中很大一部分生成、整理和分析工作，但它并不会自动替你把项目做好。

你仍然需要做三件关键的事情：

定义目标

判断结果

持续反馈

也就是说，AI 不会代替你思考，但它可以极大提高你把想法变成成果的速度。

所以，这篇文章真正想教给同学们的，不是某一个具体接口怎么写，而是下面这条完整路径：

需求 -\> 计划 -\> 开发与测试 -\> 本地演示 -\> GitHub/Gitlab 提交 -\> CI/CD上线

这条路径，也正是 Vibe Coding 在真实项目中的典型使用方式。

**二、什么是 Vibe Coding**

很多同学第一次听到这个词，会误以为它的意思是：

"把一句话丢给 AI，让 AI 帮我把所有代码写完。"

这其实是一种误解。

真正的 Vibe Coding 不是把项目完全交给 AI，而是把 AI 当作一个协作伙伴。你不需要自己包办所有细节，但你必须始终掌握项目的目标、过程和判断权。

在这个过程中：

你负责提出目标

你负责判断结果是否合理

AI 负责帮助你整理、生成、修改、解释和补全

所以，Vibe Coding 不是"少思考"，而是把更多注意力放到更重要的问题上，比如：

我到底想做什么

这个结果是不是我要的

下一步应该继续推进哪里

出现问题后应该如何修正

对于初学者来说，这种方式尤其重要，因为它降低了"必须先精通代码才能开始做项目"的门槛。

你不一定要先成为一个熟练程序员，才有资格做项目。你可以先成为一个会提问、会拆任务、会判断结果的人。

**三、为什么这个案例适合用来学习**

这次我们讨论的功能，不是一个空泛的练习题，而是一个真实项目中的实际需求：

在 NanaDraw 中加入 PDF 解析能力，让用户上传论文或技术文档后，可以从文档中选择内容来辅助绘图。

如果只用一句话概括，这个需求似乎很简单：

"给 NanaDraw 增加 PDF 上传和解析功能。"

但一旦真正开始做，同学们很快就会发现，这句话远远不够。

因为它没有说明：

PDF 从哪里上传

上传后谁来解析

解析结果展示在哪里

用户如何从中选择内容

被选中的内容如何参与后续绘图

也正因为这样，这个案例特别适合用来学习 Vibe Coding。

它能让大家看到，一个项目并不是靠一句"大而全"的指令完成的，而是靠一次次更明确的提问、更小步的推进和更具体的反馈，逐渐成型的。

**四、第 1 步：先把需求说清楚**

在 Vibe Coding 里，第一步通常不是写代码，而是写 Spec，也就是把需求描述清楚。

这一点非常关键。

因为 AI 的输出质量，往往直接取决于你的输入质量。如果你只说：

"帮我做一个 PDF 绘图功能。"

AI 也许会给你很多看起来很完整的内容，但这些内容很可能不适合当前项目，也很可能和你真正想要的效果不一致。

更好的做法，是把模糊需求拆开。

例如在这个案例里，可以拆成下面这些点：

在 AI 助手区域增加 PDF 上传入口

在设置中增加解析 Token 配置

后端调用 MinerU 完成 PDF 解析

将解析结果展示为完整文本

支持用户选中文本片段作为绘图参考

将选中的内容重新接回 NanaDraw 的绘图流程

当你把问题拆到这个粒度时，项目就从一个"很大很模糊的想法"，变成了一组可以逐项推进的小目标。

这一步看起来不像开发，但它实际上决定了后面开发会不会顺利。

可以说，Spec 不是开发之前的附属工作，它本身就是开发的一部分。

**五、第 2 步：不要急着让 AI 写代码，先让它帮你做计划**

很多初学者在使用 AI 时，最常见的问题就是太着急。

刚把需求说完，就立刻让 AI：

"请把这个功能完整实现出来。"

这样做的问题在于，AI 面对的是一个还没有被整理过的大任务。它可能会一次性给出很多内容，但这类内容常常会出现几个问题：

范围太大

落地性不强

不一定适合当前代码库

难以检查

所以，更稳妥的做法是先让 AI 帮你做计划。

例如你可以这样提问：

"这是一个已有项目，我希望增加 PDF 解析功能。请你先不要直接写代码，而是帮我输出：

功能拆分

第一版最小可行方案

实现顺序

风险点

验收标准"

这样一来，AI 在这一阶段扮演的角色就不是"代码生成器"，而是"思路整理器"。

它帮助你把一个大的目标拆成几个可执行步骤，也帮助你提前发现容易忽略的问题。

对于初学者来说，这一步尤其重要。因为当你有了一个明确计划后，项目就不再像一团乱线，而更像是一张路线图。

你知道先做什么，再做什么，也知道哪里可能出错。

这会让后面的开发过程轻松很多。

当然也可以直接使用plan模式和ai进行讨论，路径更短一些。

**六、第 3 步：用 AI 推进 Build，而不是一次做完全部开发**

进入 Build 阶段后，很多同学会再次犯一个常见错误：要求 AI 一次把所有内容做完。

但真实项目往往不是这样推进的。

更合理的方式，是分轮推进。

例如：

**第一轮：先让 AI 读项目**

这一步不是直接改代码，而是先让 AI 帮你理解代码库。

你可以问：

当前项目里和这个功能相关的文件有哪些

前端入口可能在哪

设置项可能在哪一层修改

后端文档解析应该接入哪里

这一轮的目标不是"完成功能"，而是"看清地图"。

**第二轮：只做一个小目标**

当你已经知道大概改哪里后，就不要继续提笼统要求了，而是只推进一个明确小目标。

例如：

先加 PDF 上传入口

先加设置里的 Token 字段

先完成后端解析接口

这样做的好处是，每一次改动都更容易验证，也更容易出错后回退或修正。

**第三轮：继续推进下一步交互**

等第一个小目标稳定后，再继续做下一步。

比如：

展示完整文本

让用户可以滚动查看

支持选中文本作为参考

支持右键触发绘图

这就是 Vibe Coding 里非常重要的一个思路：

**不是一次让 AI 给你最终答案，而是让它一轮一轮帮你把项目推到正确方向上。**

**七、第 4 步：不会写代码时，最重要的是学会反馈**

很多同学担心：

"我不会写代码，那我怎么判断 AI 做得对不对？"

其实，初学者最重要的能力，往往不是"会不会重写代码"，而是"会不会反馈问题"。

在真实项目里，你完全可以不用技术语言，也能给出有效反馈。

例如：

这个按钮位置不对

解析结果被截断了

我想要完整文本，不要摘要

这个按钮不要关闭，要改成收起

不要在界面里直接显示某个底层服务名字

这些话听起来很日常，但它们都很有价值，因为它们描述的是产品层面的问题，是用户真实会感受到的问题。

而 AI 很擅长把这种自然语言反馈转成具体修改。

这也是 Vibe Coding 给初学者带来的一个重要变化：

你不需要先掌握全部底层细节，才能参与项目推进。你可以先从"判断结果"和"描述问题"开始，借助 AI 一步步逼近你想要的效果。

**八、第 5 步：出现报错时，不要停下来，把报错继续交给 AI**

很多同学把报错看成失败，但在 Vibe Coding 里，报错其实是下一轮协作的起点。

当你看到报错时，不需要立刻慌张，也不需要强行自己猜原因。你可以把报错信息原样交给 AI，并让它继续分析。

例如你可以这样问：

这个报错是什么意思

这个问题更像是前端还是后端

请给我一个最小修改方案

请修复这个问题，但不要影响现有功能

这种提问方式很重要，因为它把 AI 从"生成器"变成了"调试助手"。

而调试恰恰是项目里非常消耗精力的一部分。

对于初学者来说，最关键的不是自己第一次就能完全理解所有报错，而是要知道：**遇到报错时，你仍然可以继续推进，而不是被卡住。**

**九、第 6 步：测试不是额外负担，而是 AI 可以继续参与的阶段**

很多同学误以为测试是开发完成后才额外补上的事情，或者只有很会写代码的人才能做测试。

其实不是。

测试阶段同样可以借助 AI。

在这个阶段，AI 至少可以帮助你做三类事情：

列测试清单

补测试代码

设计手工验收流程

例如当功能已经基本完成时，你可以让 AI 帮你整理：

哪些是必须成功的路径

哪些是容易失败的边界情况

前端交互要重点检查什么

后端异常要重点检查什么

这样做的好处在于，即使你还不熟悉测试框架，也能先建立一个清楚的检查思路。

测试不是为了增加负担，而是为了确认你和 AI 一起做出来的东西，是真的能用。

**十、第 7 步：本地启动与演示，也可以交给 AI 来帮你整理**

很多项目做到后期，功能其实已经差不多了，但大家还是会卡住。

原因不是代码本身，而是项目还没有真正跑起来，或者虽然跑起来了，却不知道怎么演示。

这时初学者常见的问题包括：

前端和后端分别怎么启动

依赖要先装哪些

怎么确认功能已经跑通

如果要做课堂展示，应该按什么顺序操作

这时，AI 很适合扮演"本地启动助手"和"演示助手"。

例如你可以问：

"这是一个前后端项目，请先不要讲上线流程，先告诉我：

我在本地要先启动什么

前端和后端分别用什么命令运行

我应该怎样确认功能已经跑通

如果我要录一段一分钟演示，顺序怎么安排更清楚"

这一步的意义非常大，因为它直接关系到项目能不能被别人看到。

一个项目如果只能停留在代码层面，而不能在本地顺畅演示，那么它的学习价值和展示价值都会大打折扣。

所以，本地演示绝不是附属环节，它是项目闭环中的关键一步。

**十一、第 8 步：最后用 AI 整理 GitHub 交付**

当功能做好、测试也完成、本地也可以顺畅演示后，项目还差最后一步，那就是把成果整理出来。

很多同学会在这里卡住。

不是因为不会写代码，而是因为不会整理成果。

比如：

不知道怎么写提交说明

不知道怎么写 PR 描述

不知道 README 应该补什么

不知道如何向别人解释这个项目到底做了什么

这时，AI 很适合扮演"表达助手"。

你可以让它帮助你：

根据改动生成提交说明

生成适合发往 dev 分支的 PR 标题和描述

整理 README 新增内容

输出一段适合展示项目成果的简介

这一步的本质，不是"把东西传到 GitHub 上"，而是"把成果整理成别人能看懂、能接手、能复用的形式"。

在真实协作里，写清楚和做出来同样重要。

所以到了这个阶段，AI 的角色也发生了变化。它不再主要是开发助手，而更像一个帮助你表达和整理成果的助手。

**十二、从这个案例里，我们到底学到了什么**

如果只看表面，这个案例是在讲 NanaDraw 和 MinerU 的结合。

但如果看得更深一点，它真正讲的是：

一个不太懂代码的人，怎样借助 AI 推进一个真实项目。

整条路径可以概括成下面五句话：

先把需求说清楚

再让 AI 帮你做计划

用分轮方式推进开发与测试

用反馈和报错继续驱动 AI 修正结果

最后把项目整理成可演示、可交付的成果

这就是 Vibe Coding 最值得同学们掌握的地方。

它不是让你放弃思考，也不是让你跳过学习，而是帮助你把学习和实践连接起来。

你不需要等到"什么都会了"才开始做项目。相反，很多能力正是在做项目的过程中，借助 AI 一点点建立起来的。just do it！

**十三、给同学们的三个建议**

**1. 不要怕自己不会写代码**

不会写全部代码，不代表不能做项目。

只要你能描述目标、判断结果、提出反馈，你就已经可以参与项目推进。

**2. 不要一开始就追求一步到位**

项目不是一次提问就完成的，AI 也不是一次回答就能给出最终结果。

真正有效的方式，是一轮一轮推进，一次只解决一个清楚的问题。

**3. 不要把 AI 当成"自动答案机"**

AI 最有价值的时候，不是替你跳过思考，而是帮助你更高效地思考、验证和迭代。

你越会提问、越会反馈、越会判断，它就越能帮你。

**十四、课后思考与练习**

如果你想把今天这篇文章真正变成自己的能力，可以试着完成下面几个问题：

如果让你做一个自己的 AI 项目，你会先怎样描述需求？

你会让 AI 先帮你做计划，还是直接写代码？为什么？

如果 AI 给出的结果不符合预期，你准备怎样反馈？

你准备怎样判断一个项目已经"可以演示"了？

如果你要把自己的项目提交到 GitHub，你最希望 AI 帮你整理什么内容？

也可以试着再进一步，给自己设计一个小项目，例如：

用 AI 帮你做一个课程资料整理工具

用 AI 帮你做一个文档可视化工具

用 AI 帮你做一个实验流程图生成器

然后按照本文的结构，自己走一遍：

需求 -\> 计划 -\> 开发 -\> 测试 -\> 本地演示 -\> GitHub 交付

只要你真的走完一次，这篇文章里的内容就不再只是"看懂"，而会变成你真正会用的能力。

**十五、结语**

这篇文章最想告诉同学们的一句话是：

**不会写代码，并不等于不能做项目；关键是要学会借助 AI，把想法一步步推进成成果。**
